### 第 2 页：项目背景与问题

**核心痛点：**

传统测试用例编写存在三个问题：

1. **人工编写成本高**
2. **测试场景容易遗漏**
3. **测试结果缺少统一分析**

**讲稿：**

在实际软件测试过程中，测试用例的设计往往依赖人工经验。对于登录功能来说，虽然看起来简单，但需要考虑正常登录、用户名异常、密码异常、边界值、SQL 注入、XSS 攻击等多种场景。如果完全依靠人工编写，不仅耗时，而且容易遗漏边界情况。因此，我希望借助 LLM 的能力，提高测试用例生成效率，并把生成、执行和分析流程串联起来。

---

### 第 3 页：项目目标

**目标：**

本项目希望实现一个完整的测试辅助流程：

1. 自动生成登录测试用例
2. 自动执行 Selenium 测试
3. 自动收集测试结果
4. 自动生成可视化分析大屏
5. 提供 Gradio 统一操作界面

**讲稿：**

本项目的目标不是单纯调用一次大模型，而是构建一个完整的测试辅助闭环。首先由 LLM 生成结构化测试用例，然后由 Selenium 对登录页面进行自动化测试，测试结果会被保存为 JSON 文件，最后通过 pyecharts 生成可视化大屏，帮助分析不同测试类别的分布、执行耗时和验证规则命中情况。

---

### 第 4 页：系统架构

**技术栈：**

- Python
- Conda 环境

- Gradio
- Selenium
- pytest
- pandas / openpyxl
- pyecharts

**系统流程：**

```text
DeepSeek API
   ↓
测试用例生成
   ↓
Excel / Markdown
   ↓
Selenium 自动化测试
   ↓
JSON 测试结果
   ↓
pyecharts 可视化分析
```

**讲稿：**

系统主要分为四个部分。第一部分是 Day1，用于调用 DeepSeek API 生成测试用例，并保存为 Markdown 和 Excel。第二部分是 Day2，使用 Selenium 对登录页面执行自动化测试。第三部分是 Day3，使用 Gradio 将生成测试用例和执行自动化测试整合到一个 Web 界面中。第四部分是 Day4，读取测试结果并生成 pyecharts 可视化大屏。

---

### 第 5 页：核心功能一 —— LLM 生成测试用例

**功能说明：**

Day1 根据登录规则生成测试用例，包括：

- 正常登录场景
- 用户名异常场景
- 密码异常场景
- 边界值场景
- SQL 注入测试
- XSS 攻击测试

**输出：**

- `test/day1/output/testcases.xlsx`
- `test/day1/output/testcases.md`

**讲稿：**

Day1 是测试用例生成模块。系统会根据预设的提示词，要求模型按照固定 Markdown 表格格式输出测试用例。生成内容会被解析并保存为 Excel 文件。这样可以把非结构化的模型输出转化为可管理的测试数据。

**可强调：**

这里并不是简单让模型自由生成文本，而是通过固定字段、固定表格格式和规则约束，让输出结果更容易被程序解析。

---

### 第 6 页：核心功能二 —— Selenium 自动化测试

**测试对象：**

`test/day2/login.html`

**测试工具：**

- Selenium
- pytest
- Chrome / ChromeDriver

**测试内容：**

- 正常登录
- 用户名格式异常
- 密码格式异常
- 边界值
- 安全攻击输入

**输出：**

- `test/day2/selenium/test_results.json`
- 失败截图：`test/day2/selenium/screenshots/`

**讲稿：**

Day2 是自动化测试执行模块。系统使用 Selenium 打开登录页面，自动输入用户名和密码，并点击登录按钮。每条测试都会记录测试类别、输入内容、执行状态、实际输出和耗时。如果测试失败，系统还会自动保存截图，方便后续排查问题。

**可强调：**

这个模块不只是判断测试是否通过，还会把每条测试的结果结构化保存，为后续可视化分析提供数据基础。

各位老师好，我今天汇报的项目是“基于 LLM 协同辅助自动化测试的登录测试用例生成与分析系统”。

在软件测试过程中，测试用例设计是非常重要的一环。以登录功能为例，虽然它看起来简单，但实际上需要考虑正常登录、用户名格式异常、密码格式异常、边界值、SQL 注入、XSS 攻击等多种情况。如果完全依靠人工编写测试用例，不仅耗时，而且容易遗漏一些边界场景。

因此，本项目希望借助大语言模型的能力，构建一个从测试用例生成、自动化测试执行到结果可视化分析的完整流程。

本系统主要分为四个部分。

第一部分是 Day1，也就是测试用例生成模块。它通过 DeepSeek API 调用大语言模型，根据登录规则生成结构化的测试用例。生成结果会被解析并保存为 Markdown 和 Excel 文件，方便后续查看和管理。

第二部分是 Day2，也就是 Selenium 自动化测试模块。系统会打开登录页面，自动输入用户名和密码，并执行登录操作。测试过程中会记录每条用例的测试类别、输入内容、执行状态、实际输出和耗时。如果测试失败，还会自动保存截图，方便排查问题。

第三部分是 Day3，也就是 Gradio 统一平台。它将测试用例生成、自动化测试执行、手动登录测试和结果查看整合到一个 Web 界面中。这样用户不需要在多个命令行之间切换，可以更直观地操作整个系统。

第四部分是 Day4，也就是 pyecharts 可视化分析模块。它会读取自动化测试生成的 JSON 结果，并生成可视化大屏。大屏中包括测试输入类型分布、各类别平均执行耗时、验证结果分布，以及测试类别和输入类型之间的交叉分析。

本项目的核心亮点在于形成了一个完整闭环：  
先用 LLM 生成测试用例，再用 Selenium 执行自动化测试，然后保存结构化测试结果，最后通过可视化方式分析测试效果。

在测试规则方面，系统对用户名和密码都有明确约束。用户名要求 3 到 20 位，只能包含字母、数字和下划线；密码要求 8 到 20 位，并且必须包含大小写字母和数字。同时，系统还会检测 SQL 注入和 XSS 攻击输入，避免非法输入被接受。

当然，目前项目也存在一些不足。首先，当前登录页面是教学模拟页面，没有连接真实后端数据库。其次，LLM 生成的内容仍然依赖模型输出质量。再次，目前测试规模有限，还没有接入持续集成流程。后续可以从接入真实接口、使用更严格的输出格式约束、扩展更多业务模块、接入 GitHub Actions 等方面进行改进。

总的来说，本项目实现了从测试用例生成、自动化执行到结果可视化的完整流程。它可以帮助测试人员或学习者更高效地设计测试用例，并通过自动化方式验证登录功能，最后通过可视化结果分析测试覆盖情况。

以上就是我的项目汇报，请各位老师批评指正。

## 5. 老师可能问的问题与回答

### 问题 1：为什么选择用 LLM 生成测试用例？

**回答：**

因为测试用例设计非常依赖经验，尤其是边界值、异常输入和安全测试场景，人工编写容易遗漏。LLM 可以根据规则快速生成多类型测试用例，提高测试设计效率。同时，本项目通过固定输出格式和后续解析，把模型输出转化为可执行、可管理的测试数据。

---

### 问题 2：LLM 生成的测试用例质量如何保证？

**回答：**

目前主要通过三点控制质量。第一，提示词中明确字段要求和测试场景要求；第二，要求模型按照固定 Markdown 表格格式输出；第三，程序会解析表格并保存为 Excel，后续还可以加入格式校验、去重和人工审核机制。未来可以进一步使用 JSON Schema 或更严格的输出约束提升稳定性。

---

### 问题 3：为什么使用 Selenium？

**回答：**

因为登录功能属于典型的前端交互场景，需要模拟真实用户输入用户名、密码并点击登录按钮。Selenium 可以驱动浏览器完成这些操作，因此适合用来验证登录页面在不同输入情况下的行为是否符合预期。

---

### 问题 4：测试页面为什么没有连接数据库？

**回答：**

当前项目是一个教学演示项目，重点在于展示“LLM 生成测试用例 + 自动化测试执行 + 结果可视化分析”的完整流程。登录页面中的账号是硬编码的，便于演示和验证。后续如果要用于真实项目，可以接入后端接口或数据库。

---

### 问题 5：Selenium 测试如何自动打开登录页面？

**回答：**

在 `conftest.py` 中定义了 pytest fixture，它会自动启动本地 HTTP 服务器来提供 `login.html` 页面，不需要手动启动服务器。测试时 Selenium 会访问这个本地页面并执行登录操作。

---

### 问题 6：测试结果保存在哪里？

**回答：**

Selenium 测试结束后，结果会保存到：

```text
test/day2/selenium/test_results.json
```

每条结果包含测试 ID、测试名称、测试类别、输入内容、执行状态、实际输出、截图路径和执行耗时。

---

### 问题 7：Day4 可视化图表有什么意义？

**回答：**

Day4 的图表不是简单展示测试结果，而是帮助分析测试覆盖情况。例如，可以看到正常场景、异常场景、边界值和安全测试各有多少条；也可以看到不同测试类别的平均耗时，以及实际触发了哪些验证规则。这有助于发现测试分布是否均衡。

---

### 问题 8：项目最大的难点是什么？

**回答：**

我认为难点主要有两个。第一，是 LLM 输出格式的稳定性。模型可能不会每次都严格按照要求输出，所以需要通过提示词约束和解析逻辑处理。第二，是自动化测试流程的整合。需要把用例生成、测试执行、结果收集和可视化分析连接起来，保证数据格式一致。

---

### 问题 9：如果 API 调用失败怎么办？

**回答：**

目前系统会检查 `DEEPSEEK_API_KEY` 是否存在，如果缺少密钥会给出提示。如果 API 返回空内容，也会抛出错误提示。后续可以增加重试机制、日志记录和本地缓存，提高系统稳定性。

---

### 问题 10：这个项目有什么实际应用价值？

**回答：**

这个项目可以用于登录、注册、表单校验等功能的测试辅助。它能够帮助测试人员快速生成基础测试用例，减少重复劳动，并通过自动化测试和可视化分析提高测试效率。对于教学场景来说，也能帮助学习者理解 LLM、自动化测试和可视化分析如何结合使用。
